= image:hiha-90x150.jpg[] hiha: an “orthogonal” programming language

I really, really do not like overcomplication, and so here will be
hiha, an “orthogonal” language with no formal report and just a Pratt
parser.

This compiler is a work in progress. Coding began 6 April 2026.

Do not expect development speed. The programmer is medically retired
and disabled. Also he is likely to use this project to fulfill
longstanding desires, such as to have C implementations of things he
implemented long ago in ATS.

But let me sketch some of the current doings and future directions.

I have implemented lexical analysis via Pratt parser and plugins.

First all the input files are put together in sequence, rather than
compiled separately. This is done to be contrarian. The texts are all
converted to UTF-32 and NFC canonicalized. They are _not_ NFKC
canonicalized, and thus you can use superscripts, subscripts, and so
forth, to convey meaning. But they _are_ normalized, which likely is
more than a lot of compilers and interpreters do. Thus you can rely on
“e” followed by a joining acute accent being treated exactly the same
as “é”.footnote:[Ligatures, serving as esthetic or readability
entities, rather than as linguistic entities, are not treated
similarly. The sequence “f” “i” is not converted to the “fi”
ligature.]

Then the Unicode code points of the files are individually made into
lexical tokens, and each end of file is also made into a lexical
token. However, each end of file token except the last is coverted to
a code point token for a form feed character. This is done because
form feeds are a non-line white space character that is very rare, now
that impact printers are uncommon. Thus, it was thought, no great harm
would be done by the substitution.

Thence, the tokens are run through the Pratt parser with a particular
set of rules, designated for lexical analysis, that is loaded by
plugins. The tokens that are outputted by the parser are fed back
through the the parser again and again, until a fixed point is
reached. It is expected that a fixed point will be reached _very_
quickly.

For example, on one pass, a run of digits may be collected into an
“integer” token. On the next pass an “integer” token, followed by a
decimal point, followed by an “integer”, may be collected into another
kind of token. Then, on a later pass, this token may be collected into
a “number with exponent” token.

This mechanism already is working.

***

Regarding how branching probably will work—

Almost no one makes branching nondeterministic. Programmers these days
do not even know what it means for branching to be nondeterministic.
How long has it been since Edsger W. Dijkstra’s _A Discipline of
Programming_, with its nondeterministic constructs, has been in print?
We intend to make branching in hiha nondeterministic. There will, most
likely, be two constructs, *do* and *while*. The *do* construct will
not be written “*do*...*od*” but “(*do*...)” and the *while* construct
“(*while*...)”, because hiha seeks to make things simpler than anyone
before has imagined.

What is the advantage of nondeterminism? I had to scan through my
wife’s copy of _A Discipline of Programming_ to remind myself what it
was.footnote:[I read _A Discipline of Programming_ circa 1981 from the
copy in my county college library. From 1981 to now is a long time.]
Nondeterminism lets you apply the _weakest_ precondition when proving
a snippet of code.

***

At a later time I will fill in details about how I intend to implement
a variation of threaded code, and thus attain proper tail calls
efficiently in C _without_ either trampolines or Cheney-on-the-MTA.

I will also describe how I intend to allocate memory in blocks and
then have variables that are entirely local and by reference. Variable
_names_ will then be managed in a way that substitutes for linear
_types_. Thus _syntax_ takes the place of a typechecker.

Also the type of data in hiha should probably belong to the data and
not to the variable. Thus variables probably will not need type
declarations. However, casts probably will be needed strictly. Being
an ATS programmer, I am not a big fan of loose typechecking.

I am not a young person—not ancient, but old enough to be “old.”
Should I drop dead, feel free to take over the project. Or not. I
might take over such a thing from someone, but that does not mean
someone else would.

Please try to understand why I am strict about structured programming
in C. It is related to why my C functions tend to be very small.
People who are not strict about structured programming in a language
like C soon find themselves losing track of how the program actually
works. You have to actually use structured programming to see what I
mean.

Do not bother arguing with detractors. They are being unscientific,
arguing without first making observations. I am asking you to observe.

You may program however you wish, in a project you have taken over.
When I took over ANT and made kompostilo typesetter from it, I
converted from Revised Syntax for OCaml to standard syntax, simply so
I could use Emacs Tuareg mode. But I recommend using my style. You may
like it, should you actually try it. C is a difficult language to be
disciplined in. OpenBSD tries to be disciplined in it by using only a
restricted set of functions, and by flinging insults at everyone else.
They write good code but theirs is not my approach. I try to be
disciplined in C by using “ancient” methods of disciplined
programming—and by keeping the cyclometric complexity not only to a
minimum, but visually obvious.footnote:[Also, one must admit, there is
the experience of writing *memcpy* calls that satisfied proofs in ATS
code.]
